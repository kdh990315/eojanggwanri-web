# PRD: 어장관리 웹사이트 MVP

## 문제 정의

초보 낚시인은 낚시 중 처음 보는 물고기를 잡았을 때 이름을 바로 알고 싶지만, 검색어를 어떻게 입력해야 할지 모르거나 이미지 검색 결과를 신뢰하기 어렵다. 어장관리는 이 사용자가 검색으로 웹사이트에 유입된 순간, 사진으로 어종 후보를 확인하고 그 결과를 조과 기록이라는 더 큰 가치로 자연스럽게 이어가야 한다.

현재 구현은 소개 페이지 중심의 프로토타입 초안이다. 제품 기준, SEO 기준, AI 어종 판별 흐름, 광고와 개인정보 처리 기준을 정식 PRD로 묶어 현재 구현을 검수하고 이후 vertical slice 이슈로 나눌 기준이 필요하다.

## 해결 방향

어장관리 웹사이트는 Next.js App Router 기반의 한국어 앱 소개/AI 어종 판별 웹사이트로 구축한다. 메인 페이지는 어장관리 앱 소개를 위한 페이지로 유지한다. 메인 페이지 안의 AI 어종 판별 섹션에는 실제 판별 페이지로 이동하는 버튼을 제공하되, 사진 업로드나 분석 UI는 넣지 않는다. 실제 사진 업로드와 판별 체험은 별도 AI 어종 판별 페이지에서만 제공한다.

사용자는 검색 유입 후 메인 페이지의 AI 어종 판별 소개 섹션 또는 어종별 SEO 페이지에서 `AI 어종 판별하기` 버튼을 통해 AI 어종 판별 페이지로 이동한다. AI 어종 판별 페이지에서 사진을 업로드하면 영상 광고가 1회 표시되고, 사용자가 skip하거나 광고 시간이 끝나면 기존 Supabase Edge Function을 통해 AI 어종 후보를 확인한다. 결과는 확정값이 아니라 참고용 후보로 표시하며, 판별한 데이터를 보여주는 결과 화면에서 어장관리 앱의 조과 기록 가치를 다시 한 번 소개한다.

## 사용자 시나리오

1. 웹 방문자는 메인 페이지에서 어장관리 앱의 조과 기록 가치를 먼저 이해하기를 원한다. 이를 통해 이 사이트가 단순 판별 도구가 아니라 낚시 기록 앱 소개라는 점을 알 수 있다.
2. 검색으로 유입된 초보 낚시인은 AI 어종 판별 소개 섹션에서 분석 페이지로 이동하는 명확한 버튼을 원한다. 이를 통해 소개 페이지 안에서 업로드 UI를 찾지 않고 `/ai-species`로 이동할 수 있다.
3. 검색으로 유입된 초보 낚시인은 물고기 사진으로 이름을 확인할 수 있는 진입점을 원한다. 이를 통해 검색어를 몰라도 다음 행동을 바로 선택할 수 있다.
4. 모바일 사용자는 모바일에서 누르기 쉬운 AI 어종 판별 CTA를 원한다. 이를 통해 현장에서 빠르게 판별 페이지로 이동할 수 있다.
5. 웹 방문자는 메인 페이지가 전체 판별 UI를 포함하지 않는 구조를 원한다. 이를 통해 소개 페이지와 실제 판별 페이지의 역할을 명확히 구분할 수 있다.
6. 초보 낚시인은 사진 업로드 전 AI 결과가 후보라는 안내를 원한다. 이를 통해 판별 결과를 확정값으로 오해하지 않는다.
7. 초보 낚시인은 흐린 사진, 부분 사진, 여러 마리 사진의 한계를 안내받기를 원한다. 이를 통해 결과 품질을 현실적으로 이해할 수 있다.
8. 비로그인 사용자는 로그인 없이 AI 어종 판별을 사용할 수 있기를 원한다. 이를 통해 앱 가입 전에 즉시 문제를 해결할 수 있다.
9. 비로그인 사용자는 판별 사진이 저장되거나 보관되지 않는다는 명확한 안내를 원한다. 이를 통해 개인정보 처리 방식을 이해하고 사용할 수 있다.
10. 개인정보에 민감한 사용자는 사진 처리와 보관 기준이 실제 구현과 일치하는 설명을 원한다. 이를 통해 서비스 신뢰도를 판단할 수 있다.
11. AI 어종 판별 사용자는 사진을 선택한 뒤 영상 광고가 표시되는 흐름을 원한다. 이를 통해 AI 판별 전에 광고 노출 단계를 이해할 수 있다.
12. AI 어종 판별 사용자는 광고를 skip하거나 광고 시간이 끝나면 판별이 시작되는 흐름을 원한다. 이를 통해 광고 클릭 없이 다음 단계로 진행할 수 있다.
13. AI 어종 판별 사용자는 광고 실패가 앱 오류처럼 보이지 않는 대체 안내를 원한다. 이를 통해 막힌 느낌 없이 판별 흐름을 이해할 수 있다.
14. AI 어종 판별 사용자는 판별 중 로딩 상태를 원한다. 이를 통해 이미지가 처리되고 있음을 알 수 있다.
15. AI 어종 판별 사용자는 판별 실패와 재시도 상태를 원한다. 이를 통해 네트워크나 이미지 문제 후 다시 시도할 수 있다.
16. AI 어종 판별 사용자는 후보 어종명과 참고 신뢰도 표현을 원한다. 이를 통해 어떤 어종일 가능성이 있는지 판단할 수 있다.
17. AI 어종 판별 사용자는 판별 결과를 직접 확인하고 수정할 수 있다는 안내를 원한다. 이를 통해 AI 결과를 최종 선택으로 오해하지 않는다.
18. AI 어종 판별 사용자는 다른 사진으로 다시 판별하는 기능을 원한다. 이를 통해 더 나은 사진으로 결과를 개선할 수 있다.
19. 앱 잠재 사용자는 AI 판별 결과 화면에서 어장관리 앱을 다시 소개받기를 원한다. 이를 통해 사진 판별 이후 조과 기록 앱을 써야 하는 이유를 이해할 수 있다.
20. 조과 기록 사용자는 날짜, 위치, 물때, 날씨, 마릿수, 사진을 함께 남기는 가치 설명을 원한다. 이를 통해 물고기 이름 확인 이상의 기록 가치를 이해할 수 있다.
21. 조과 기록 사용자는 고급 통계가 MVP 이후 확장 가치로 표현되는 설명을 원한다. 이를 통해 현재 제공 기능과 미래 방향을 구분할 수 있다.
22. 앱 탐색 사용자는 실제 사용할 기능 위주의 앱 기능 요약을 원한다. 이를 통해 커뮤니티, 랭킹, 결제 같은 비핵심 기능에 혼란스럽지 않다.
23. 앱 설치 관심 사용자는 소개 페이지에서만 앱 다운로드 링크를 확인하기를 원한다. 이를 통해 AI 판별 페이지에서는 판별과 결과 확인 흐름에 집중할 수 있다.
24. 앱 설치 관심 사용자는 스토어 링크 준비 전 더미 상태가 명확한 안내를 원한다. 이를 통해 사용할 수 없는 다운로드 버튼을 실제 링크로 오해하지 않는다.
25. 어종 검색 사용자는 우럭, 광어, 백조기, 놀래미 같은 어종별 진입 페이지를 원한다. 이를 통해 특정 어종 검색 의도에서 AI 판별로 이동할 수 있다.
26. 어종 검색 사용자는 어종별 페이지에서 생태, 금어기, 금지체장 정보가 DB 기준으로 표시되기를 원한다. 이를 통해 부정확한 정보를 단정적으로 받지 않는다.
27. 어종 검색 사용자는 유사 어종과 혼동 가능성을 안내받기를 원한다. 이를 통해 AI 후보와 실제 어종을 더 신중하게 비교할 수 있다.
28. FAQ 방문자는 정확도, 사진 저장, 광고, 조과 기록 연동에 대한 답변을 원한다. 이를 통해 사용 전 불안을 줄일 수 있다.
29. 문의 사용자는 문의, 오류 제보, 아이디어 제안 링크를 원한다. 이를 통해 서비스 개선 의견을 전달할 수 있다.
30. 검색엔진 사용자는 각 페이지가 하나의 검색 의도에 맞는 title, description, H1을 갖는 구조를 원한다. 이를 통해 검색 결과에서 필요한 페이지를 쉽게 찾을 수 있다.
31. 접근성 사용자는 키보드 포커스, 스크린 리더 라벨, 구체적인 이미지 대체 텍스트를 원한다. 이를 통해 보조 기술로도 주요 흐름을 사용할 수 있다.
32. 운영자는 Gemini API key, Supabase service role key, 광고 서버 secret이 클라이언트에 노출되지 않는 구조를 원한다. 이를 통해 보안 사고를 방지할 수 있다.
33. 운영자는 기존 Supabase Edge Function 재사용을 원한다. 이를 통해 앱과 웹의 AI 판별 로직을 중복 구현하지 않는다.
34. 운영자는 비로그인 사용량에 대한 파일 크기, MIME type, rate limit, 악성 파일 방어 기준을 원한다. 이를 통해 공개 웹 체험의 abuse risk를 줄일 수 있다.
35. 개발자는 현재 프로토타입을 첫 vertical slice 결과물로 매핑할 수 있는 기준을 원한다. 이를 통해 되돌리지 않고 부족한 부분만 이슈 기준으로 다듬을 수 있다.

## 구현 결정

- Next.js App Router를 기준으로 구현한다.
- 화면 문구, SEO title, SEO description, 본문은 한국어를 사용한다.
- URL slug는 영어를 사용한다.
- 메인 페이지는 어장관리 앱 소개 페이지다.
- 메인 페이지의 AI 어종 판별 섹션은 분석 페이지로 이동시키는 소개 섹션이며, 업로드 UI나 결과 UI를 포함하지 않는다.
- 메인 페이지에는 전체 AI 판별 UI를 구현하지 않는다.
- AI 어종 판별 섹션의 버튼 문구는 `AI 어종 판별하기`이며 AI 어종 판별 페이지로 이동한다.
- AI 어종 판별 페이지는 실제 어종 분석만 담당하며 초기 안내, 이미지 업로드, 영상 광고, 광고 skip 또는 시간 종료, AI 판별 로딩, 결과, 실패, 재시도 상태를 가진다.
- AI 어종 판별 결과는 확정값이 아니라 후보와 참고 정보로 표시한다.
- AI 어종 판별 결과 화면은 판별한 후보 데이터를 보여주면서 어장관리 앱의 조과 기록 가치를 다시 소개한다.
- confidence 또는 신뢰도 표현은 모델 참고 지표로만 다루며 정확도 보장으로 표현하지 않는다.
- 웹 AI 어종 판별은 로그인 없이 사용할 수 있다.
- 비로그인 사용자의 판별 사진은 저장하거나 보관하지 않는다.
- 기술적으로 임시 처리가 필요하면 저장된 사용자 콘텐츠로 노출하지 않고 처리 후 제거하는 기준을 둔다.
- AI 어종 판별은 기존 Supabase Edge Function을 재사용한다.
- Gemini API key는 Supabase Edge Function secret으로 관리하고 웹 클라이언트 번들, public 환경 변수, 클라이언트 컴포넌트에 노출하지 않는다.
- Supabase service role key와 광고 서버 secret도 웹 클라이언트에 노출하지 않는다.
- 클라이언트는 이미지 선택, 업로드 또는 요청 오케스트레이션, 화면 상태 전환만 담당한다.
- AI 요청에는 최소한의 이미지 데이터, 이미지 경로, 사용자 또는 세션 식별자만 포함한다.
- 사용자의 정확한 낚시 포인트 위치는 반드시 필요하지 않으면 AI prompt에 포함하지 않는다.
- 광고는 보상형 광고가 아니다.
- 이미지 업로드 후 영상 광고를 노출하고, 사용자가 skip하거나 광고 시간이 끝나면 AI 판별을 시작한다.
- 광고 클릭을 유도하지 않는다.
- 광고 플랫폼, SDK, script tag, 서버 검증 방식은 아직 미정이며 구현 전 최신 공식 문서를 확인한다.
- 앱 다운로드 링크는 소개 페이지에서만 노출한다.
- AI 어종 판별 페이지에는 앱 다운로드 링크를 직접 노출하지 않고, 필요하면 소개페이지의 앱 소개 또는 다운로드 섹션으로 연결한다.
- 스토어 링크가 준비되기 전에는 준비 중 상태와 더미 UI임을 명확히 표시한다.
- 문의, 오류 제보, 아이디어 제안은 Google Form으로 받는다.
- 소개 페이지에는 GitHub 링크를 노출하지 않는다.
- 어종 생태, 금어기, 금지체장 정보는 Supabase DB 데이터를 사용한다.
- 어종별 SEO 페이지는 대표 이미지, AI 판별 CTA, 조과 기록 안내, 유사 어종 혼동 가능성, DB 기반 생태와 규제 정보를 포함한다.
- 어종별 대표 이미지의 출처와 라이선스는 아직 미정이며 구현 전 결정한다.
- FAQ 페이지는 AI 정확도, 사진 저장, 광고 노출, skip 이후 동작, 조과 기록 저장, 웹 단독 사용, 잘못된 결과 수정 가능성을 다룬다.
- SEO는 각 페이지가 하나의 검색 의도에 집중하도록 구성한다.
- 각 페이지는 title, description, canonical, Open Graph title, Open Graph description, Open Graph image를 정의한다.
- 각 페이지는 H1을 하나만 가진다.
- FAQ 페이지는 FAQPage JSON-LD를 사용한다.
- 어종별 페이지는 WebPage 또는 Article JSON-LD를 사용한다.
- 소개 페이지는 앱 다운로드 링크가 준비된 뒤 SoftwareApplication JSON-LD 적용을 검토한다.
- JSON-LD 내용은 화면에 보이는 내용과 일치해야 한다.
- 디자인은 흰색 기반 미니멀 UI, 선명한 검정 텍스트, 얇은 회색 구분선, 제한적인 바다색 포인트를 사용한다.
- CTA, 링크 hover, 선택 상태 외에 바다색을 과하게 사용하지 않는다.
- 과한 파도 그래픽, 장식성 배경, 어두운 해양 테마는 기본 방향으로 삼지 않는다.
- 접근성 기준으로 키보드 포커스, 업로드 input 라벨, 구체적인 이미지 alt, 오류 메시지의 다음 행동 안내를 포함한다.
- 현재 구현된 소개 페이지 프로토타입은 첫 vertical slice인 `소개 페이지 MVP 구현` 결과물 후보로 둔다.
- 현재 프로토타입은 이후 이슈 기준으로 CTA 경로, 라우트 존재 여부, 섹션 구성, 금지 표현, 다운로드 더미 상태, SEO 메타데이터를 검수한다.

## 테스트 결정

- 테스트는 내부 구현 세부사항이 아니라 사용자가 관찰하는 외부 동작을 기준으로 작성한다.
- CTA 테스트는 AI 어종 판별 섹션의 `AI 어종 판별하기` 버튼이 AI 어종 판별 페이지로 이동하는지 검증한다.
- 소개 페이지 테스트는 Hero, 조과 기록 가치, 앱 기능 요약, AI 어종 판별 소개, 자주 찾는 어종 링크, FAQ 요약, 하단 CTA가 노출되는지 검증한다.
- 소개 페이지 테스트는 AI 어종 판별 소개 섹션의 `AI 어종 판별하기` 버튼이 `/ai-species`로 이동하는지 검증한다.
- 문구 테스트는 AI 결과가 확정값이 아니라 후보와 참고 정보로 표현되는지 검증한다.
- 금지 표현 테스트는 100% 정확도, AI 확정, 자동 저장, 광고 클릭 유도, 보상형 광고 오해 문구가 화면에 없는지 검증한다.
- AI 어종 판별 상태 테스트는 초기 안내, 이미지 선택, 광고 표시, skip 또는 시간 종료, 로딩, 결과, 결과 내 앱 가치 재소개, 실패, 재시도 상태 전환을 검증한다.
- 광고 흐름 테스트는 광고 클릭 없이 skip 또는 시간 종료 후 AI 판별이 시작되는지 검증한다.
- 사진 처리 테스트는 비로그인 사용자의 사진이 저장 또는 보관되지 않는 계약을 검증한다.
- Edge Function 호출 테스트는 Gemini API key가 클라이언트에 노출되지 않고 서버 경계에서만 사용되는지 검증한다.
- 업로드 보안 테스트는 파일 크기 제한, 이미지 MIME type 검증, 악성 파일 방어, rate limit 정책을 검증한다.
- 어종별 페이지 테스트는 Supabase DB 데이터가 없을 때 단정적인 생태, 금어기, 금지체장 정보를 표시하지 않는지 검증한다.
- FAQ 테스트는 필수 질문과 답변 원칙이 화면과 JSON-LD에 일치하는지 검증한다.
- SEO 테스트는 각 라우트의 metadata title, description, canonical, Open Graph 값, 단일 H1을 검증한다.
- 접근성 테스트는 업로드 input의 접근 가능한 이름, 버튼 포커스 상태, 이미지 alt, 오류 메시지의 다음 행동 안내를 검증한다.
- 현재 코드베이스에는 테스트 선례가 아직 없으므로 첫 이슈에서는 최소한 TypeScript typecheck와 빌드를 기준 검증으로 삼고, 이후 UI 테스트 도구를 도입할 때 사용자 흐름 중심으로 확장한다.
